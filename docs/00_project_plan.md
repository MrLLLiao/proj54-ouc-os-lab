# 项目计划

## 总目标

建设一套面向中国海洋大学低年级计算机学生的 OS 竞赛入门实验体系。项目暂定以 xv6-riscv 为主线，形成可阅读、可复现、可验证、可扩展的 lab0-lab5 实验链路，并保留教程、代码框架、测试用例、参考实现、FAQ、AI 使用记录、引用说明和过程日志。

当前阶段不追求完整 OS 实现，而是优先形成符合 proj54 教学型赛题要求的项目方案和可持续迭代结构。

## 每周验收标准

- 有清晰的可复核产物，而不是口头进展。
- 所有未完成内容标注 TODO、待验证或计划中。
- 真实运行过的命令必须记录环境、命令、输出位置和结论。
- 未运行的命令不得写成“已通过”。
- 每周至少更新一次 docs/06_progress_log.md。

## 提交节奏

- 每 1-2 天至少一次有效 commit。
- 文档、脚本、测试、代码尽量分开提交。
- commit message 建议使用 `type: summary`，例如：
  - `docs: refine MVP plan and lab0 lab1 documentation`
  - `scripts: add minimal lab entry helpers`
  - `tests: document lab1 test plan`
- 删除平台占位文件的提交说明应包含：`replace platform placeholder with project scaffold`。
- 提交前至少运行 `git diff --check`；涉及脚本时运行相应脚本。

## 风险与止损策略

| 风险 | 触发信号 | 止损策略 |
| --- | --- | --- |
| 范围过大 | lab 数量多但没有真实可运行链路 | 优先保证 lab0/lab1 质量，推迟 lab3-lab5 深化 |
| 环境问题拖慢进度 | 多台机器无法一致安装工具链 | 先固定 Windows 11 + WSL2 Ubuntu 推荐环境，并保留问题记录 |
| baseline 引入不规范 | 许可证或来源未记录 | 暂停源码引入，先补齐 references 和 license |
| AI 生成内容未经复核 | 文档中出现夸大、虚假或不清楚来源的内容 | 必须人工复核，AI 使用记录标注影响范围 |
| 测试不足 | 只有文档没有真实执行证据 | 将测试报告模板和命令输出记录作为阶段验收项 |
| 时间不足 | 第 3 周仍未完成 lab1 闭环 | 缩小范围，优先交付 lab0/lab1 和高质量报告 |
